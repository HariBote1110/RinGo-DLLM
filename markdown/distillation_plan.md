# 蒸留計画書: Qwen 3.5 9B → RinGo-DLLM 55M

## 概要

オープンウェイトの Qwen 3.5 9B を教師モデルとして、RinGo-DLLM（55M Masked Diffusion LM）に
ロジット蒸留を行い、限られたデータ・計算資源で言語モデルの品質を飛躍的に向上させる。

---

## 背景・動機

### 現状の課題

| 指標 | 現在値 | 目標 |
|---|---|---|
| val_loss | ~5.78（v2 Epoch 1） | < 4.0 |
| パープレキシティ | ~324 | < 50（文法的に自然な文が出る目安） |
| 生成品質 | 高頻度語の羅列 | 文脈に合った文を生成 |

WikiText-103 での通常学習は改善が続いているが、55M モデル × 1 億トークンという組み合わせでは
**データから学べる情報量に上限がある**。蒸留により、同じデータから遥かに豊かな学習信号を得られる。

### なぜ蒸留が効くのか

通常の学習（ハードラベル）:
```
正解: "Paris"  → one-hot ベクトル [0, 0, ..., 1, ..., 0]
→ 学習信号: 「Paris が正解」（1 ビットの情報）
```

蒸留（ソフトラベル）:
```
教師の出力: P("Paris")=0.85, P("Lyon")=0.05, P("Marseille")=0.03, ...
→ 学習信号: 「Paris が最有力だが、Lyon や Marseille も都市として妥当」
→ 語彙間の意味的関係を暗黙的に学べる（dark knowledge）
```

同一バッチで**数千倍の情報量**が得られるため、小さいモデルでも品質が大幅に向上する。

## アーキテクチャ設計

### Phase 1: 教師ロジットの事前生成

学習ループ内でリアルタイムに教師推論すると GPU を奪い合うため、
**ロジットを事前計算してディスクに保存**する方式を採用する。

```
WikiText-103 チャンク（128 トークン）
        ↓ BertTokenizer でトークナイズ
   input_ids (B, 128)
        ↓ Qwen 3.5 9B に入力
   teacher_logits (B, 128, 151936)   ← 巨大！
        ↓ BERT 語彙 30,522 にマッピング
   teacher_logits_mapped (B, 128, 30522)
        ↓ Top-K=64 のみ保存（容量削減）
   teacher_topk.pt  (B, 128, 64) × 2（indices + values）
```

### Phase 2: 語彙マッピング（Qwen → BERT）

Qwen と BERT はトークナイザが異なるため、語彙のマッピングが必要。

**アプローチ: テキスト経由のソフトマッピング**

```python
# 1. Qwen の各語彙をデコードしてテキスト化
# 2. そのテキストを BERT トークナイザで再エンコード
# 3. Qwen vocab → BERT vocab のマッピングテーブルを構築
#    (1:1 でない場合はサブワードの平均 or 最尤マッピング)
```

完全に一致しない語彙は捨てるか、近似マッピングで対応する。
実装の複雑さに比べて効果が限定的であれば、**データ蒸留（テキスト生成のみ）に
フォールバック**する選択肢も残す。

### Phase 3: 蒸留学習

```python
# 損失関数
loss = α * CE_loss(student_logits, hard_labels)      # 通常のクロスエントロピー
     + β * KL_loss(student_logits, teacher_logits)    # 教師分布との KL ダイバージェンス

# 推奨ハイパーパラメータ（初期値）
α = 0.3    # ハードラベルの重み
β = 0.7    # ソフトラベルの重み
temperature = 4.0  # ソフトラベルの温度（高いほど分布が平滑化）
```

## 実装計画

### Step 1: ロジット事前生成スクリプト（`distill/generate_logits.py`）

- Qwen 3.5 9B を RTX 3070 Ti にロード（量子化: Q8 or Q4）
- WikiText-103 の全チャンクを入力し、Top-K ロジットを保存
- 推定所要時間:
  - 889,969 チャンク × 128 トークン ÷ 65 t/s ≈ **約 49 時間**
  - ただしバッチ推論で高速化可能（2〜4 倍）

**ディスク容量の見積もり（Top-K=64）:**
```
indices: 889,969 × 128 × 64 × 2 bytes (int16) ≈ 13.8 GB
values:  889,969 × 128 × 64 × 2 bytes (float16) ≈ 13.8 GB
合計: ~28 GB
→ C ドライブの空き容量に注意！外部ストレージ or D ドライブ推奨
```

### Step 2: 語彙マッピング（`distill/vocab_mapping.py`）

- Qwen トークナイザの全語彙をテキスト化
- BERT トークナイザで再エンコード
- マッピングテーブルを JSON で保存
- カバレッジ率を計測（目標: 90% 以上）

### Step 3: 蒸留学習スクリプト（`distill/train_distill.py`）

- 既存の `train.py` をベースに拡張
- 事前生成ロジットをロードしてバッチごとに教師信号として使用
- α, β, temperature は CLI 引数で調整可能に
- Discord Webhook 通知対応

### Step 4: 評価

- 蒸留前後の val_loss / パープレキシティ比較
- `sample.py` での生成品質の比較
- CoreML 変換 + ANE ベンチマーク（推論性能は変わらないはず）

## 代替アプローチ: データ蒸留（ロジット蒸留が困難な場合）

語彙マッピングが複雑すぎる場合、シンプルなデータ蒸留にフォールバックする。

```
Qwen 3.5 9B に文頭を与える → 続きを生成 → 大量のテキストを収集
→ そのテキストを BERT トークナイザで再エンコード
→ 通常の CE_loss で学習（ただし教師が生成した高品質テキスト）
```

**利点:**
- 語彙マッピング不要
- 実装が単純
- WikiText-103 以外の多様なテキストを生成できる

**欠点:**
- ロジット（dark knowledge）が得られない
- ロジット蒸留に比べて効率が低い

## Discord チャットログの活用（データ蒸留の seed データ）

### 概要

5年分の個人 Discord チャットログを蒸留用プロンプトの **seed データ** として活用する。
ログをそのまま学習データにするのではなく、Qwen 3.5 9B による拡張・匿名化パイプラインを通すことで、
プライバシーを保護しつつ多様で自然なテキストコーパスを生成する。

### パイプライン

```
Discord チャットログ（5年分）
    ↓ エクスポート（DiscordChatExporter 等）
    ↓ 前処理
raw_messages.jsonl
    ↓ フィルタリング
      - 短すぎる発言を除去（< 10 文字）
      - スタンプ・リアクション・URL のみの行を除去
      - 連続する短文を会話ターンとして結合
    ↓ 匿名化
      - ユーザー名・サーバー名・チャンネル名をプレースホルダに置換
      - 電話番号・メールアドレス等の PII を除去
    ↓ Qwen 3.5 9B による拡張
      - 会話の文頭やトピックを seed として Qwen に続きを生成させる
      - 口語 → 書き言葉へのリライトも可能
    ↓ 品質フィルタリング
      - パープレキシティベースで低品質テキストを除去
      - 重複排除（MinHash）
expanded_corpus.jsonl
    ↓ BERT トークナイザでエンコード
    ↓ 通常の CE_loss で蒸留学習
```

### 期待されるメリット

| メリット | 詳細 |
|---|---|
| **トピックの多様性** | 技術議論、日常会話、ゲーム、音楽など 5 年分の幅広いジャンル |
| **自然な文体** | 教科書的でない口語・くだけた表現が含まれる |
| **プライバシー保護** | Qwen による拡張で元の発言が直接含まれない |
| **コスト効率** | 既に手元にあるデータを活用、追加のデータ取得不要 |

### 注意事項

- 現在の RinGo-DLLM は BERT トークナイザ（英語中心）のため、日本語ログの活用には将来的なトークナイザ変更が前提
- 英語での会話ログがあればそのまま活用可能
- 他の参加者の発言を seed に使う場合は、匿名化を徹底すること
- データ量の目安: 5 年 × 100 msg/日 ≈ 18 万メッセージ → Qwen 拡張後に数百万トークン規模が期待できる

## リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| 語彙マッピングのカバレッジ不足 | ロジット蒸留の効果が限定的 | カバレッジ 80% 以下ならデータ蒸留にフォールバック |
| ロジット保存の容量不足 | C ドライブ枯渇 | D ドライブ or 外部ストレージに保存 |
| 教師推論の所要時間 | 2 日以上かかる可能性 | バッチ推論 + Q4 量子化で高速化 |
| WSL2 クラッシュ | ロギット生成が中断 | チャンク単位で逐次保存、再開可能に |
| 温度パラメータの調整 | 蒸留品質への影響大 | T=2, 4, 8 で比較実験 |
